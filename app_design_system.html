KeelWorks Typography

KeelWorks typography provides a clear, accessible reading experience across products by pairing a readable body typeface with a distinctive heading typeface and a consistent scale.

Body: Lato Headings: Raleway Base: 18px Scale: 1.125 (Major Second) Goal: Clarity + hierarchy + accessibility

Principles

Clarity

Prioritize readability and scanability in all contexts, from dense dashboards to long-form content.

  • Prefer simple, open letterforms and comfortable spacing.
  • Keep headings short; use body text for detail.

Consistency

Use a shared scale and token set so typography feels unified across platforms and surfaces.

  • Always use tokens instead of ad-hoc sizes.
  • Maintain hierarchy order (don’t skip levels).

Hierarchy

Create obvious structure with size, weight, and spacing—without relying on color alone.

  • Use Display styles for page and section titles.
  • Use Heading styles for component and content headings.

Accessibility

Support users with low vision, cognitive load constraints, or reading challenges.

  • Keep sufficient size and line-height for body text.
  • Meet WCAG contrast requirements for text and labels.

Type System

The system uses Lato for body/UI text and Raleway for headings/display, balancing readability with emphasis.

Heading Typeface: Raleway

  • Display and Heading tokens.
  • Titles and short-to-medium phrases.
  • Avoid for dense paragraphs.

Body Typeface: Lato

  • Body, Label, and Caption tokens.
  • Long reading and UI copy.
  • Generous line-height for legibility.
Rule of thumb: Raleway communicates structure. Lato communicates content.

We use an 18px base and the Major Second scale (1.125) for predictable steps that preserve hierarchy and accessibility.

Usage Guidelines

Use typography to communicate structure first, then style. Follow these rules to keep pages readable and consistent.

Do

  • Use display-* for pages and major sections.
  • Use heading-* for content and component headings.
  • Use body-md as the default for paragraph text.
  • Keep headings short; move details into body text.
  • Maintain hierarchy (don’t jump from Display to small labels).

Don’t

  • Use Raleway for dense paragraphs or long-form reading.
  • Skip levels (e.g., display-xl → caption).
  • Invent one-off font sizes that bypass tokens.
  • Reduce line-height below approved specs to “fit more.”
  • Rely on color alone for hierarchy.

Platform Notes

  • Desktop web: Prefer body-md or body-lg depending on density.
  • Mobile: Keep headings at or below heading-lg unless intentionally editorial.
  • Data-heavy UIs: Use body-sm sparingly; ensure contrast and spacing remain comfortable.

Mobile

KeelWorks products run on mobile. Typography should stay readable at arm’s length, adapt to smaller viewports, and avoid overly large display styles that push content below the fold.

Recommended defaults

  • Default body: body-md (18px). Use body-sm only for dense UI where space is constrained.
  • Headings: Prefer heading-md / heading-lg. Avoid display-xl on small screens.
  • Line length: Keep paragraphs short; break long instructions into bullets.

Suggested responsive behavior

Use one of these patterns and apply it consistently:

Pattern A — Same tokens, usage limits

  • Keep token specs identical on all platforms.
  • On mobile, cap headings at heading-lg and use display-lg only for top-level screens.

Pattern B — Responsive token overrides

  • Use the same token names, but adjust a few sizes at a defined breakpoint (e.g., ≤768px).
  • Typical reductions: display-xl → 32px, display-lg → 28px, heading-xl → 22px.

Breakpoints (documentation)

  • Mobile: ≤ 768px (or your app’s “compact” width class)
  • Tablet: 769–1024px
  • Desktop: ≥ 1025px
If the product uses native dynamic type (iOS) or font scaling (Android), tokens should map to the system’s text styles and respect user font-size settings.

Accessibility

Typography must support legibility for all users. Follow WCAG guidance for contrast and keep token sizing unless a documented exception exists.

Minimum guidance

  • Prefer body-md (18px) or larger for default reading.
  • Use caption only for supporting content and with strong contrast.
  • Don’t reduce line-height below token specs.
  • Ensure interactive text meets contrast requirements in all states.
If you need an exception (e.g., dense tables), document the reason and add compensating support (contrast, spacing, zoom).

Implementation Notes

Keep tokens consistent between design and code. Minimal CSS token example:

:root {
  --font-body: "Lato", system-ui, -apple-system, Segoe UI, Roboto, Helvetica, Arial;
  --font-heading: "Raleway", system-ui, -apple-system, Segoe UI, Roboto, Helvetica, Arial;

  --display-xl-size: 36px; --display-xl-lh: 44px; --display-xl-wt: 600;
  --display-lg-size: 32px; --display-lg-lh: 38px; --display-lg-wt: 600;
  --display-md-size: 28px; --display-md-lh: 34px; --display-md-wt: 500;

  --heading-xl-size: 26px; --heading-xl-lh: 33px; --heading-xl-wt: 500;
  --heading-lg-size: 24px; --heading-lg-lh: 31px; --heading-lg-wt: 500;
  --heading-md-size: 22px; --heading-md-lh: 28px; --heading-md-wt: 500;
  --heading-sm-size: 20px; --heading-sm-lh: 26px; --heading-sm-wt: 500;

  --body-lg-size: 20px; --body-lg-lh: 30px; --body-lg-wt: 400;
  --body-md-size: 18px; --body-md-lh: 27px; --body-md-wt: 400;
  --body-sm-size: 16px; --body-sm-lh: 24px; --body-sm-wt: 400;
  --label-sm-size: 14px; --label-sm-lh: 21px; --label-sm-wt: 400;
  --caption-size: 12px; --caption-lh: 18px; --caption-wt: 400;
}

Recommended conventions

  • Use token names as the source of truth.
  • Map tokens to platform styles (CSS, components, native).
  • Document responsive overrides centrally.
